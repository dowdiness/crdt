# Canopy

コードを書いているとき、あなたの頭の中にはプログラムの「意味」がある。この変数が何を指しているか、この関数が何を返すか、このコードがどう動くか。

でもエディタはそれを何も知らない。ただの文字列としてコードを扱っている。

Canopyは、書いた瞬間からプログラムの意味を追いかけるエディタだ。構文木がリアルタイムに構築され、変数のスコープが解決され、式の評価結果が表示され、コードが自動的に整形される——すべてが、1文字タイプするたびに更新される。

テキストと構文木は双方向に同期する。テキストを編集すれば構造が追従し、構造を操作すればテキストが書き換わる。複数人での同時編集もサーバー不要で動作する。

特定の言語に縛られていない。文法を与えれば、この体験がそのまま手に入る。

## もう少し詳しく

### あなたが毎日やっている翻訳

コードを書くとき、あなたは常に翻訳をしている。

頭の中には「この関数は引数を2倍にする」という理解がある。でもそれをエディタに伝えるには、`λx. x + x` という文字の並びに変換しなければならない。リファクタリングするときは、コードを目で読んで頭の中に意味を復元し、変更を考え、また文字列に変換して書き戻す。

この翻訳を、あなたは一日に何百回も繰り返している。

バグを探すときも同じだ。コードをじっと見て、頭の中で実行をシミュレートし、「ここでこの変数はこの値になるはず」と推論する。エディタはその推論を一切手伝わない。変数の値も、型も、スコープも、あなたの頭の中にしかない。

この翻訳コストは、プログラミングそのものの一部だと思われている。でも本当は、道具の限界がもたらしているものだ。ロジックを考えることは難しくない。それを何も理解していないテキストエディタを経由して表現することが、苦しいのだ。

### 書くだけで、構造が生まれる

Canopyでは、書く行為と構造化が同時に起きる。

文字をタイプした瞬間に、エディタはプログラムの構造を理解している。「解析」ボタンを押すのでも、保存して結果を待つのでもない。書くという行為そのものが、構造を生み出す。

あなたは何も整理していない。ただ書いただけだ。でもエディタの中には、あなたが書いたコードの構造と意味が、すでに存在している。

この「書くだけで構造が生まれる」という感覚は、一度体験すると元に戻れなくなる。従来のエディタが、文字列を保存するだけの道具に見えてくる。

### テキストからも、構造からも

普通のエディタでは、コードに触れる手段はテキスト編集だけだ。関数の引数を入れ替えたければ、文字列をカット＆ペーストする。式を関数に抽出したければ、自分でテキストを切り出して新しい関数を書き、元の場所を呼び出しに書き換える。

Canopyでは、テキストとして編集してもいいし、構造として操作してもいい。どちらで変更しても、もう一方が自動的に追従する。

これは「機能が二つある」ということではない。見方が二つある、ということだ。同じプログラムを、テキストとしても、構造としても見ることができる。そして、今の作業に合った見方で操作できる。テキストの帳尻合わせ——括弧の対応、インデントの調整、参照の更新——はエディタが引き受ける。

あなたは「何をしたいか」だけを伝えればいい。

### コードの先にあるもの

Canopyがコードエディタとして解いている問題——書いたものを自動的に構造化し、意味を追跡し、必要な情報を表示する——は、コードに限った話ではない。メモ、アイデア、URL、読書記録。あらゆる「書く」行為に同じ課題がある。

コードエディタは、この体験の最初の実証だ。その先には、書くだけで知識が構造化され、必要なときに必要な文脈が浮かび上がるシステムがある。詳しくは[プロダクトビジョン](architecture/product-vision.md)を参照。

### 距離を縮める

プログラミングが難しいのは、あなたの意図とコンピュータの理解の間に距離があるからだ。「こうしたい」を伝えるために、意味を構造に、構造を構文に、構文を文字の列に——何段階もの変換を経なければならない。

Canopyは、この距離を一段ずつ縮めていく。文字列を構造にする。構造に意味を付ける。意味をあなたの目に見える形にする。

道具を使っていることを意識しなくなる瞬間がある。手に持ったペンの存在を忘れて、ただ書いている。楽器を弾いていて、指の動きではなく音楽そのものに没入している。コンピュータでその瞬間を作ること——それがCanopyの目指すところだ。
